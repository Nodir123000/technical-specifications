# ⏰ Skill: deadline-watchdog

**ID:** `deadline-watchdog`
**Версия:** 1.0
**Владелец:** Process Agent
**Стандарт:** O'z DSt 1987:2018 §6.5.3 — Контроль сроков исполнения

---

## Описание

Автоматическое отслеживание сроков исполнения поручений, ревизий и документов. Рассылает эскалационные уведомления по схеме T-7, T-3, T-0. При просрочке инициирует протокол эскалации.

---

## Входные параметры

```json
{
  "check_type": "enum: [SINGLE, BULK, SCHEDULED]",
  "items": [
    {
      "item_id": "string (UUID)",
      "item_type": "enum: [AUDIT, DOCUMENT, ORDER, FINDING]",
      "deadline": "date (YYYY-MM-DD)",
      "responsible_user_id": "string (UUID)",
      "escalation_user_id": "string (UUID)",
      "priority": "enum: [LOW, MEDIUM, HIGH, CRITICAL]"

  "current_date": "date (для тестирования — иначе NOW())"

```

## SLA

| Метрика | Значение |
|:---|:---|
| Время проверки пакета (≤1000 элементов) | ≤ 2000 мс |
| Расписание проверки | Каждый день в 08:00 (UTC+5) |
| Гарантия доставки уведомления | ≥ 99% |

---

## Схема уведомлений

| Порог | Тип уведомления | Получатель |
|:---|:---|:---|
| T-7 (7 дней до дедлайна) | Информационное | Ответственный |
| T-3 (3 дня до дедлайна) | Предупреждение | Ответственный + CHIEF_INSPECTOR |
| T-0 (день дедлайна) | Срочное | Ответственный + CHIEF_INSPECTOR |
| T+1 (просрочка) | КРИТИЧЕСКОЕ + эскалация | Ответственный + Руководство |
| T+3 и далее | Ежедневная эскалация | Руководство + ADMIN |

---

## Preconditions

- Запускается по расписанию (cron) или вручную администратором
- Таблица `deadlines` содержит актуальные записи

## Postconditions

- Все уведомления записаны в `notification_log`
- Просроченные элементы помечены флагом `overdue=true` в БД
- Метрика `overdue_count` обновлена в Monitoring Agent

---

## Error Flows

| Ошибка | Действие |
|:---|:---|
| Ошибка отправки уведомления | Повтор 3 раза, затем запись в `failed_notifications` |
| БД недоступна | Пропустить цикл, зафиксировать пропуск в лог, повтор через 30 мин |
| Ответственный пользователь заблокирован | Направить уведомление напрямую `escalation_user_id` |

---

## Тест-кейсы

### TC-01: Уведомление T-3

**Вход:** Ревизия с дедлайном через 3 дня
**Ожидание:** Уведомление отправлено ответственному и CHIEF_INSPECTOR

### TC-02: Просроченный элемент

**Вход:** Документ просрочен на 2 дня
**Ожидание:** `overdue_days: 2`, эскалация на руководство, флаг в БД

### TC-03: Массовая проверка

**Вход:** 500 элементов с разными дедлайнами
**Ожидание:** Завершение ≤ 2000 мс, корректная группировка уведомлений
