# 🔐 Skill: access-control

**ID:** `access-control`
**Версия:** 1.0
**Владелец:** Security Agent
**Стандарт:** O'z DSt 2814:2019 §4 — Управление доступом; RBAC-матрица (Приложение З)

---

## Описание

Реализация ролевой модели доступа (RBAC) согласно Матрице разрешений АИС КРУ МО РУЗ. Проверяет, разрешена ли конкретная операция для конкретной роли на конкретном ресурсе. Является единственным источником правды о правах доступа.

**Ссылка:** [RBAC_Matrix.md](../Матрица_связей/RBAC_Matrix.md)

---

## Входные параметры

```json
{
  "user_id": "string (UUID)",
  "roles": ["array of enum: [ADMIN, CHIEF_INSPECTOR, INSPECTOR, VIEWER, AUDITOR, ...]"],
  "operation": "enum: [C, R, U, D, A, X]",
  "resource": "string (например: 'audits', 'violations', 'audit_log')",
  "resource_owner_id": "string (UUID, для проверки 'своё')",
  "context": {
    "document_status": "string (опционально — для условных прав)",
    "is_own_resource": "boolean"
  }
}
```

---

## Выходные параметры

```json
{
  "granted": "boolean",
  "effective_role": "string (роль, давшая доступ)",
  "restriction": "string | null (например: 'только черновик', 'только своё')",
  "policy_ref": "string (ссылка на пункт матрицы)"
}
```

---

## SLA

| Метрика | Значение |
|:---|:---|
| Время проверки | ≤ 5 мс (кэшированные роли) |
| Кэш ролей | Инвалидируется при изменении назначения роли |
| Доступность | 99.99% |

---

## Логика проверки (по RBAC_Matrix.md)

```text

1. Получить текущие роли пользователя из кэша (или БД)

2. Для каждой роли проверить: разрешена ли операция на ресурсе

3. Применить контекстные ограничения ('только черновик', 'только своё')

4. Проверить SoD (разделение обязанностей) — §З.7.1

5. Проверить временные ограничения (период действия роли) — §З.7.2

6. Вернуть результат

```

---

## Специальные правила (SoD)

| Проверка | Условие блокировки |
|:---|:---|
| Создатель ≠ Утверждающий | `creator_id == approver_id` → отказ |
| INSPECTOR + AUDITOR | Обе роли у одного пользователя → отказ |
| Период роли истёк | `role_expires_at < NOW()` → отказ |

---

## Preconditions

* Пользователь прошёл аутентификацию

* Роли пользователя актуальны в кэше

## Postconditions

* Результат проверки закэширован на 5 минут (для повторных запросов)

* Каждый отказ записывается в `security_incidents` (для анализа)

---

## Error Flows

| Ошибка | Действие |
|:---|:---|
| Роли пользователя не найдены | `granted: false` — безопасный отказ |
| Кэш недоступен | Читать напрямую из БД (с предупреждением о задержке) |
| Неизвестный ресурс | `granted: false`, запись в лог с уровнем WARN |

## Тест-кейсы

### TC-01: INSPECTOR создаёт черновик

**Вход:** roles=[INSPECTOR], operation=C, resource=audits, document_status=null
**Ожидание:** `granted: true`, `restriction: null`

### TC-02: VIEWER пытается создать документ

**Вход:** roles=[VIEWER], operation=C, resource=audits
**Ожидание:** `granted: false`, `policy_ref: "§З.3.4"`

### TC-03: INSPECTOR утверждает свой документ (SoD)

**Вход:** roles=[INSPECTOR], operation=A, resource=audit_reports, is_own_resource=true
**Ожидание:** `granted: false` (нарушение SoD — создатель не может утверждать)

### TC-04: Истёкшая роль

**Вход:** roles=[CHIEF_INSPECTOR], role_expires_at=2026-01-01
**Ожидание:** `granted: false`, уведомление HR + ADMIN
