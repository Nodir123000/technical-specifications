# 📋 Skill: logging-diagnostics

**ID:** `logging-diagnostics`
**Версия:** 1.0
**Владелец:** Monitoring Agent
**Стандарт:** O'z DSt 2814:2019 §7 — Ведение журналов событий

---

## Описание

Централизованный навык логирования событий системы и самодиагностики. Обеспечивает структурированное ведение журналов аудита, системных событий и ошибок. Данные используются для соответствия требованиям O'z DSt 2814 и для расследования инцидентов.

---

## Входные параметры

```json
{
  "log_type": "enum: [AUDIT, SYSTEM, ERROR, SECURITY, PERFORMANCE]",
  "level": "enum: [DEBUG, INFO, WARN, ERROR, CRITICAL]",
  "event": {
    "source_agent": "string",
    "action": "string",
    "user_id": "string (UUID, опционально)",
    "resource_type": "string",
    "resource_id": "string",
    "result": "enum: [SUCCESS, FAILURE, PARTIAL]",
    "details": "object (без ПИНФЛ и паролей)",
    "duration_ms": "number"
  },
  "timestamp": "datetime (ISO 8601)"
}
```

---

## Выходные параметры

```json
{
  "log_id": "string (UUID)",
  "stored": "boolean",
  "storage_target": "enum: [DB, FILE, BOTH]",
  "retention_days": "number"
}
```

## SLA

| Метрика | Значение |
|:---|:---|
| Время записи лога | ≤ 10 мс (асинхронно) |
| Доступность | 99.99% |
| Срок хранения audit_log | 5 лет (требование O'z DSt 2814) |
| Срок хранения system_log | 1 год |
| Срок хранения error_log | 3 года |

---

## Правила логирования

| Уровень | Когда использовать |
|:---|:---|
| `DEBUG` | Только в dev/test окружении, отключён в prod |
| `INFO` | Успешные операции, штатные события |
| `WARN` | Нестандартные ситуации без ошибки |
| `ERROR` | Ошибки, операция не выполнена |
| `CRITICAL` | Ошибки безопасности, сбои критических компонентов |

## Запрещённые данные в логах

- ПИНФЛ в открытом виде → заменять на `[MASKED]`
- Пароли и токены → `[REDACTED]`
- Полные номера банковских счетов → `[MASKED]`

---

## Preconditions

- Вызывается асинхронно (не блокирует основной поток)
- БД логов (`audit_log`, `system_log`) доступна

## Postconditions

- Запись доступна для чтения немедленно
- CRITICAL-записи дублируются в файловую систему (резерв)
- Истёкшие логи автоматически архивируются по расписанию

---

## Error Flows

| Ошибка | Действие |
|:---|:---|
| БД логов недоступна | Запись в резервный файл `/var/log/ais-kru/fallback.log` |
| Ошибка маскирования | Лог не записывается, генерируется `MASKING_FAILED` alert |
| Диск заполнен | Алерт CRITICAL администратору, удаление старых DEBUG-логов |

---

## Тест-кейсы

### TC-01: Запись AUDIT-события

**Вход:** Пользователь создал документ, уровень INFO
**Ожидание:** `stored: true`, `log_id` заполнен, ПИНФЛ не присутствует в записи

### TC-02: CRITICAL-событие

**Вход:** Попытка эскалации привилегий, уровень CRITICAL
**Ожидание:** Запись в БД + в файл, немедленный алерт администратору

### TC-03: Fallback при недоступности БД

**Имитация:** БД логов возвращает ошибку подключения
**Ожидание:** Запись в `/var/log/ais-kru/fallback.log`, `storage_target: FILE`
