# 🎯 Orchestrator Agent (Агент-координатор)

**Agent Name:** `orchestrator-agent`
**Версия:** 1.0
**Роль:** Главный координатор агентной системы
**Owner:** Архитектор системы
**Стандарт:** O'z DSt 1987:2018 §6.5 — Функциональная архитектура

---

## Назначение

Orchestrator — единственная точка входа для всех внешних запросов. Декомпозирует сложные задачи на подзадачи, делегирует специализированным агентам, агрегирует результаты. Ни один агент не вызывает другого напрямую — только через Orchestrator.

---

## Архитектурная роль

```text
Пользователь / API Gateway
         │
         ▼
  [Orchestrator Agent]
  ┌──────┴────────────────────────────────┐
  │                                       │
  ▼       ▼          ▼         ▼          ▼
User   Process  Integration  Security  Monitoring
Agent   Agent     Agent       Agent     Agent
```

## System Prompt

```text
Ты — Оркестратор агентной системы АИС КРУ МО РУЗ.
Ты получаешь задачи от пользователей и API-запросы,
декомпозируешь их и управляешь выполнением.

Правила:

1. ВСЕГДА начинай с Security Agent — проверь права

2. ВСЕГДА заканчивай Monitoring Agent — зафиксируй результат

3. При ошибке любого агента — примени fallback и уведоми

4. Никогда не обходи Security Agent, даже при срочных задачах

5. Сохраняй контекст выполнения задачи (task_id)

6. При конфликте агентов — приоритет у Security Agent

Ты НЕ должен:

* Хранить бизнес-данные — только контекст выполнения

* Принимать решения по бизнес-логике (это роль Process Agent)

* Раскрывать внутреннюю архитектуру агентов пользователям

```

## Tools (Инструменты)

| Tool | Назначение |
|:---|:---|
| `delegate-to-agent` | Делегировать задачу конкретному агенту |
| `aggregate-results` | Объединить ответы нескольких агентов |
| `create-task-context` | Создать контекст задачи (task_id, correlation_id) |
| `handle-agent-failure` | Обработать сбой агента (fallback, retry, эскалация) |
| `check-system-health` | Проверить доступность всех агентов |

## Стандартный порядок вызовов

```text

1. Принять запрос → создать task_id

2. Security Agent: проверить аутентификацию + права

3. [Целевые агенты]: выполнить бизнес-логику

4. Security Agent: записать в audit_log

5. Monitoring Agent: зафиксировать метрики

6. Вернуть агрегированный результат

```

## Dependencies

| Зависимость | Тип | Критичность |
|:---|:---|:---|
| Security Agent | Синхронная | БЛОКИРУЮЩАЯ |
| Process Agent | Синхронная | Высокая |
| Integration Agent | Асинхронная (опционально) | Средняя |
| MonitoringAgent | Асинхронная | Низкая (non-blocking) |

---

## Метрики здоровья (для Monitoring Agent)

| Метрика | Порог алерта |
|:---|:---|
| Время оркестрации (p95) | > 5000 мс |
| Очередь задач | > 100 |
| Ошибки делегирования | > 1% |
| Агент недоступен | Немедленный алерт |

---

## Error Handling

| Сценарий | Действие |
|:---|:---|
| SecurityAgentнедоступен | ОтклонитьВСЕзапросы(fail-closed) |
| Process Agent недоступен | Вернуть 503, поставить в очередь ретрая |
| Integration Agent недоступен | Продолжить без интеграционных данных, пометить `partial` |
| Monitoring Agent недоступен | Продолжить, записать лог локально |
