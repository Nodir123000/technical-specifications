# UC-02: Выявление и регистрация нарушения

**ID:** UC-02
**Приоритет:** Критический
**Акторы:** Инспектор (INSPECTOR), Главный инспектор (CHIEF_INSPECTOR)
**Агенты:** Orchestrator, Security, Process, User
**Связанные тест-кейсы:** TC-VIOL-001 — TC-VIOL-010
**Зависит от:** UC-01 (ревизия должна быть создана и иметь статус IN_PROGRESS)

---

## Precondition

- Ревизия существует со статусом `IN_PROGRESS`
- INSPECTOR назначен на данную ревизию
- Данные организации получены (из UC-01)

---

## Main Flow

```
1.  INSPECTOR → UI: Открыть ревизию → "Добавить нарушение"
2.  User Agent → Security Agent: Проверить права CREATE для findings
3.  Security Agent → User Agent: ✅ Разрешено (§З.3.4)
4.  User Agent → INSPECTOR: Форма регистрации нарушения
5.  INSPECTOR → UI: Заполнить (тип, сумма, ссылка на документ, описание)
6.  User Agent → Process Agent: Передать данные нарушения
7.  Process Agent → data-validation: Валидация (schema: violation.create)
8.  data-validation → Process Agent: ✅ Данные корректны
9.  Process Agent → kru-inspection-expert: Оценить нарушение
    Вход: {audit_id, transactions, inspector_id, inspection_type}
10. kru-inspection-expert → Process Agent:
    {verdict: VIOLATIONS_FOUND, violations: [...], total_damage_uzs: X}
11. Process Agent → БД: Сохранить нарушение (статус: Зафиксировано)
12. Process Agent → calculate-damages: Пересчитать общий ущерб по ревизии
13. Process Agent → Security Agent: audit_log (FINDING_CREATED)
14. Process Agent → User Agent: ✅ Нарушение зарегистрировано
15. User Agent → INSPECTOR: Отобразить обновлённый список нарушений
16. Process Agent → deadline-watchdog: Зарегистрировать дедлайн устранения
```

---

## Alternative Flow A: Низкий confidence_score

```
10a. kru-inspection-expert → Process Agent:
     {requires_human_review: true, confidence_score: 0.65}
10b. Process Agent → notify: Уведомить CHIEF_INSPECTOR о необходимости проверки
10c. Process Agent → БД: Сохранить нарушение с флагом needs_review=true
10d. User Agent → INSPECTOR: "Нарушение требует проверки главного инспектора"
```

## Alternative Flow B: Превышает порог существенности

```
11b. Если total_damage_uzs > 500_000_000 UZS (порог существенного ущерба):
11c. Process Agent → notify: CRITICAL-уведомление руководству
11d. Process Agent → Security Agent: Записать событие SIGNIFICANT_DAMAGE
```

---

## Postcondition

- Нарушение сохранено в БД
- Общий ущерб по ревизии пересчитан
- Дедлайн устранения зарегистрирован
- При needs_review=true — задача для CHIEF_INSPECTOR создана

---

## Human-in-the-loop точки

| Момент | Действие | Роль |
|:---|:---|:---|
| При needs_review=true | Ручная проверка нарушения | CHIEF_INSPECTOR |
| При significant_damage | Согласование дальнейших действий | Руководство |
