# UC-05: Аварийный сценарий — Недоступность Security Agent

**ID:** UC-05
**Приоритет:** КРИТИЧЕСКИЙ (обязателен для госприёмки)
**Акторы:** Система (автоматический), Дежурный администратор
**Агенты:** Orchestrator, Monitoring
**Связанные тест-кейсы:** TC-ERR-001 — TC-ERR-008

---

## Описание

Сценарий описывает поведение системы при полной недоступности Security Agent — критичнейшего компонента. Демонстрирует принцип FAIL-CLOSED и протокол аварийного восстановления.

---

## Precondition

* Система работала штатно

* Security Agent стал недоступен (сбой, перезагрузка, сетевой сбой)

---

## Main Flow (Обнаружение и реакция)

```text

1. Monitoring Agent: check-metrics (каждые 5 минут)

2. Monitoring Agent → Security Agent: health-check

3. Security Agent: ❌ Нет ответа (таймаут 3000 мс)

4. Monitoring Agent: Повтор health-check через 30 секунд

5. Security Agent: ❌ Снова нет ответа

6. Monitoring Agent → create-alert:

    {level: CRITICAL, type: "SECURITY_AGENT_DOWN", duration: "60s"}

1. Monitoring Agent → notify-oncall:

    Немедленное уведомление дежурного (телефон + Telegram)

1. Orchestrator: Обнаруживает недоступность Security Agent

2. Orchestrator → FAIL-CLOSED: Отклонить ВСЕ входящие запросы

    Всем пользователям: HTTP 503 "Система временно недоступна"

1. Monitoring Agent → logging-diagnostics:

    Записать инцидент в security_incidents (тип: SERVICE_DOWN)
```

---

## Main Flow (Восстановление)

```text

1. Дежурный администратор: Получает уведомление

2. Администратор: Диагностирует и устраняет причину

3. Security Agent: Перезапущен и отвечает на health-check

4. Monitoring Agent → Orchestrator: Security Agent восстановлен

5. Orchestrator: Снять FAIL-CLOSED режим

6. Orchestrator: Отправить запрос к Security Agent (проверка готовности)

7. Security Agent: ✅ Готов к работе

8. Monitoring Agent → create-alert:

    {level: INFO, type: "SECURITY_AGENT_RECOVERED", downtime_minutes: X}

1. Monitoring Agent → notify-oncall: Уведомление о восстановлении

2. Система: Возобновить приём запросов

3. Monitoring Agent → logging-diagnostics: Закрыть инцидент

```

---

## Временные параметры восстановления

| Этап | Целевое время |
|:---|:---|
| Обнаружение сбоя | ≤ 90 секунд |
| Уведомление дежурного | ≤ 2 минуты |
| Начало диагностики | ≤ 5 минут |
| Восстановление (RTO) | ≤ 30 минут |

---

## Альтернативный сценарий: Плановое обслуживание

```text
Перед обслуживанием:

1. ADMIN активирует режим "Maintenance Window"

2. Orchestrator: Завершить активные сессии с предупреждением

3. Security Agent: Остановить (плановая остановка)

После обслуживания:

1. Security Agent: Запустить

2. Monitoring Agent: Проверить готовность

3. Orchestrator: Открыть систему

```

---

## Postcondition

* Security Agent восстановлен и доступен

* Инцидент закрыт в `security_incidents` с указанием причины и времени простоя

* Дежурный получил уведомление о восстановлении

* Отчёт об инциденте доступен ADMIN и CHIEF_INSPECTOR

---

## Почему этот сценарий критичен для госприёмки

Регулятор проверяет: *"Что происходит с ПДн при сбое системы защиты?"*
Правильный ответ — FAIL-CLOSED: система останавливается, данные не утекают.
Неправильный ответ — FAIL-OPEN: система продолжает работу без проверки прав.
