## М.1. ОБЩИЕ ПОЛОЖЕНИЯ

### М.1.1. Назначение

Настоящее приложение определяет план миграции данных из существующих систем и источников в АИС КРУ МО РУЗ, включая анализ источников, маппинг данных, процедуры трансформации и валидации.

### М.1.2. Принципы миграции

1. **Целостность** — все данные должны быть перенесены без потерь
2. **Валидация** — каждый этап сопровождается проверкой
3. **Откат** — возможность возврата к исходному состоянию
4. **Прозрачность** — документирование каждого шага
5. **Минимальное время простоя** — миграция без остановки системы

## М.4. ПРОЦЕДУРЫ ТРАНСФОРМАЦИИ

### М.4.1. Правила трансформации данных

| Тип данных | Источник | Правило | Пример |
|-----------|---------|---------|--------|
| **Даты** | Различные | Приведение к ISO 8601 | 01.01.2024 → 2024-01-01 |
| **Числа** | XLSX | Decimal(18,2) | 1 000 000,00 → 1000000.00 |
| **Строки** | Все | TRIM, нормализация | «  текст  » → «текст» |
| **ФИО** | Разные | Приведение к формату «Фамилия И.О.» | Иванов Иван Иванович → Иванов И.И. |
| **Коды** | Excel | Uppercase | «а123» → «А123» |

### М.4.2. Правила валидации

| Правило | Описание | Действие при нарушении |
|---------|---------|----------------------|
| **NOT NULL** | Обязательные поля не пустые | Отклонение записи |
| **UNIQUE** | Уникальность ключевых полей | Дубликат → обновление |
| **FOREIGN KEY** | Ссылочная целостность | Создание placeholder |
| **FORMAT** | Соответствие формату | Трансформация |
| **RANGE** | Значение в допустимом диапазоне | Отклонение с логированием |

### М.4.3. Дедупликация

| Метод | Применение | Результат |
|-------|----------|----------|
| **Exact match** | Полное совпадение всех полей | Объединение |
| **Fuzzy match** | Похожие значения (ФИО) | Ручной просмотр |
| **Hash match** | MD5/SHA хеш | Группировка |

## М.6. ИНСТРУМЕНТЫ И СКРИПТЫ

### М.6.1. ETL-процесс

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Extract    │ ──▶ │ Transform  │ ──▶ │   Load     │
│             │     │            │     │            │
│  Источники  │     │  Правила   │     │  АИС КРУ   │
│  данных     │     │  Трансформ.│     │  PostgreSQL│
└─────────────┘     └─────────────┘     └─────────────┘
```

### М.6.2. Пример ETL-скрипта (Python)

```python
# Миграция справочника регионов
import pandas as pd
from prisma import Prisma

def migrate_regions():
    # Extract
    df = pd.read_excel('source/regions.xlsx')

    # Transform
    df['name_ru'] = df['name'].apply(lambda x: x.strip())
    df['name_uz'] = df['name_uz'].apply(lambda x: x.strip() if pd.notna(x) else None)
    df['code'] = df['code'].apply(lambda x: x.upper().strip())
    df['status'] = 'active'

    # Load
    db = Prisma()
    db.connect()

    for _, row in df.iterrows():
        db.ref_regions.upsert(
            where={'code': row['code']},
            data={
                'code': row['code'],
                'name': {'ru': row['name_ru'], 'uz': row['name_uz']},
                'status': row['status']
            }
        )

    db.disconnect()

if __name__ == '__main__':
    migrate_regions()
```

## М.7. ВАЛИДАЦИЯ И ТЕСТИРОВАНИЕ

### М.7.1. Виды проверок

| Проверка | Метод | Критерий успеха |
|----------|-------|---------------|
| **Подсчёт записей** | COUNT | Источник = Целевая |
| **Сумма числовых полей** | SUM | Расхождение < 0.01% |
| **Проверка ссылок** | FK validation | 0 нарушений |
| **Проверка формата** | Regex | 100% соответствия |
| **Ручная выборка** | Случайная выборка 1% | 100% корректности |
| **CRC-валидация** | Контрольная сумма | CRC source = CRC target |

### М.7.2. Чек-лист валидации

- [ ] COUNT (источник) = COUNT (целевая)
- [ ] SUM (amount source) = SUM (amount target) ± 0.01%
- [ ] Все FOREIGN KEY ссылаются на существующие записи
- [ ] Все обязательные поля (NOT NULL) заполнены
- [ ] Даты в диапазоне 2000-01-01 … 2030-12-31
- [ ] Коды справочников уникальны
- [ ] ПИНФЛ проходят валидацию (14 цифр)

## М.8. РОЛЛБЭК ПРОЦЕДУРА

### М.8.1. Стратегия отката

| Этап | Стратегия | Время отката |
|------|----------|-------------|
| До миграции | Полный бэкап БД + CRC | 30 мин |
| После каждой фазы | Инкрементальный бэкап + CRC | 10 мин |
| После миграции | Точка восстановления (PITR) | 5 мин |

### М.8.2. Процедура отката (обновлённая)

```python
def migration_rollback():
    """
    Процедура отката миграции при сбое.
    Включает проверку целостности бэкапа и CRC-валидацию.
    """
    # 1. НЕМЕДЛЕННО остановить ETL-процесс
    stop_etl_process()

    # 2. Проверить целостность бэкапа
    backup_checksum = calculate_sha256(backup_file)
    stored_checksum = read_stored_checksum("backup.sha256")

    if backup_checksum != stored_checksum:
        alert_administrators("CRITICAL: Backup integrity check FAILED")
        abort("Cannot proceed with rollback - backup corrupted")

    # 3. Восстановить БД из бэкапа
    restore_database(backup_file)

    # 4. CRC-валидация после восстановления
    restored_crc = calculate_table_crc()
    if restored_crc != original_crc:
        alert_administrators("CRITICAL: Restored data CRC mismatch")
        abort("Data integrity not guaranteed after restore")

    # 5. Уведомить команду
    send_alert("ROLLBACK COMPLETED: $(date)", channel="all")

    # 6. Зафиксировать инцидент
    log_incident(rollback_reason, affected_records)

    # 7. Запустить ETL повторно только после устранения причины
    wait_for_fix()
```

### М.8.3. Чек-лист перед миграцией (обязательный)

- [ ] **Обязательный бэкап** перед началом миграции
- [ ] **Фиксация CRC** исходных данных (sha256sum)
- [ ] **Проверка бэкапа** на целостность
- [ ] **Описание сценария отката** для каждой фазы
- [ ] **Доступность off-site хранилища** для бэкапов
- [ ] **Контакты команды** для экстренной связи
