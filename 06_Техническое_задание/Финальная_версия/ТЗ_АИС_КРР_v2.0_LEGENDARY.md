## НАЗНАЧЕНИЕ И ЦЕЛИ СОЗДАНИЯ СИСТЕМЫ

### 2.1. Назначение системы

АИС КРР — это **специализированная информационная система закрытого контура**, предназначенная для комплексной автоматизации деятельности Контрольно-ревизионного управления МО РУз. Система обеспечивает:

* **Дистанционный аудит**: анализ финансово-хозяйственной деятельности воинских частей без выезда инспектора на место, на основе загружаемой первичной документации и данных из ведомственных информационных систем.

* **Инспекционный контроль**: автоматизированный сбор, структурирование и документирование результатов выездных проверок с генерацией юридически значимых актов ревизий.

* **Мониторинг исполнения**: контроль сроков и качества устранения выявленных нарушений с системой автоматических эскалаций.

* **Аналитику и отчётность**: формирование управленческой отчётности, KPI-дашбордов и предиктивных моделей риска для руководства МО РУз.

### 2.2. Цели создания и измеримые результаты

|№|Цель|KPI / Метрика успеха|Базовое значение|Целевое значение|
|---|---|---|---|---|
|Ц-1|Повышениеэффективностиконтроля|Долярисковыхобъектов,охваченныхдистанционнымаудитом|0%|≥80%|
|Ц-2|Ускорениециклаобработкиакта|Среднийцикл«выявление→предписание»|20рабочихдней|≤10рабочихдней|
|Ц-3|Обеспечениенеизменяемостиактов|Доляинцидентовнесанкционированногоизменения|Неизмеряется|0%(техническиневозможно)|
|Ц-4|Централизацияданныхонарушениях|НаличиеединогореестранарушенийиМТС|Нет|Есть|
|Ц-5|Снижениебумажногодокументооборота|Доляактов,сформированныхвэлектронномвиде|<10%|≥95%|

### 2.3. Класс системы

АИС КРР относится к классу **автоматизированных систем управления** (АСУ) ведомственного уровня. По характеру обрабатываемой информации — **конфиденциальная информация (ДСП)**, класс защищённости **КЗ-2** согласно классификации МО РУз.

## ТРЕБОВАНИЯ К СИСТЕМЕ

### 4.1. Требования к структуре и функционированию

Система строится по **многоуровневой клиент-серверной архитектуре** с чётким разделением на:

```text
┌─────────────────────────────────────────────────┐
│                  Слой представления              │
│         (Next.js Web UI + PWA для выездов)       │
├─────────────────────────────────────────────────┤
│                  Слой бизнес-логики              │
│       (Node.js / NestJS REST + GraphQL API)      │
├─────────────────────────────────────────────────┤
│          Слой аналитики и ИИ-агентов             │
│      (Python FastAPI микросервисы, очереди)      │
├─────────────────────────────────────────────────┤
│                  Слой данных                     │
│   PostgreSQL (основная БД) + MinIO (документы)  │
│      Redis (кэш + сессии) + Elasticsearch        │
└─────────────────────────────────────────────────┘
```text
Все компоненты **разворачиваются в изолированной ведомственной сети** на серверах МО РУз. Взаимодействие компонентов — только через внутреннюю сеть (LAN/VLAN), без выхода в Интернет.

### 4.6. Требования к информационной безопасности (КЗ-2)

#### 4.6.1. Физическая и сетевая изоляция

* Система **разворачивается исключительно в Air-gapped сети** МО РУз. Физическое подключение к сетям общего пользования (Интернет, Intranet gov.uz) запрещено.

* Все серверные компоненты размещаются в **выделенном защищённом серверном помещении** с ограниченным физическим доступом.

* Обмен данными с объектами контроля осуществляется только через **защищённые съёмные носители** (согласно регламенту МО) или выделенные защищённые каналы.

#### 4.6.2. Аутентификация и авторизация

|Требование|Детализация|
|---|---|
|Аутентификация|Двухфакторная:логин/пароль+физическийтокен(TOTP)илисмарт-карта|
|Минимальная длина пароля|12 символов (буквы верх/нижн. регистра, цифры, спецсимволы)|
|Блокировка аккаунта|После 5 неудачных попыток входа на 30 минут|
|Сессия|Автоматическое завершение сессии при бездействии более 15 минут|
|Ролевая модель|RBAC с принципом минимальных привилегий (см. Приложение В)|

#### 4.6.3. Шифрование

|Объект|Метод|
|---|---|

#### 4.6.4. Audit Log (неизменяемый системный журнал)

Каждое событие системы обязательно записывается в **отдельную защищённую таблицу** `audit_log` с атрибутами:

|Атрибут|Тип|Описание|
|---|---|---|
|`id`|UUID|Глобально уникальный идентификатор события|
|`timestamp`|TIMESTAMPTZ|Точное время события (UTC)|
|`user_id`|UUID|Идентификатор пользователя|
|`user_role`|VARCHAR|Роль в момент действия|
|`ip_address`|INET|IP-адресрабочейстанции|
|`action`|VARCHAR|Тип действия (LOGIN, VIEW, CREATE, UPDATE, DELETE, APPROVE, EXPORT)|
|`entity_type`|VARCHAR|Тип объекта (ACT, PREDPISANIE, USER, RISK_SIGNAL)|
|`entity_id`|UUID|Идентификатор объекта|
|`old_value`|JSONB|Значение ДО изменения|
|`new_value`|JSONB|Значение ПОСЛЕ изменения|
|`hash_prev`|CHAR(64)|SHA-256хэшпредыдущейзаписи(блокчейн-цепочка)|

Таблица `audit_log` доступна только для INSERT. UPDATE и DELETE запрещены на уровне СУБД (row-level security + отдельная роль БД). Целостность цепочки хэшей проверяется автоматически раз в сутки.

### 4.7. Требования к архитектуре и технологическому стеку

Полный список компонентов — в **Приложении Д**. Ключевые ограничения:

* **Запрещено** использование проприетарного ПО без лицензии, согласованной с МО РУз.

* **Запрещено** использование SaaS-сервисов, cloud-провайдеров (AWS, GCP, Azure).

**Масштабируемость аналитики**: архитектура должна предусматривать возможность подключения внешнего аналитического хранилища (например, ClickHouse) как вторичного слоя для OLAP-запросов без изменения бизнес-логики подсистем.

* База данных должна корректно функционировать при объёме записей в таблицах до **10 000 000 строк** без деградации производительности.

* Хранилище документов (MinIO) должно масштабироваться добавлением дисковых узлов.

### А.2. Алгоритм расчёта Risk Score

Risk Score (RS) рассчитывается по формуле:

```text
RS = Σ (Wi × Ii) × 100

Где:
  Wi — вес индикатора (сумма весов = 1)
  Ii — нормализованное значение индикатора [0..1]

Индикаторы и веса:
  I1: Отклонение факт/план расходов          W1 = 0.25
  I2: Аномалии в госзакупках                 W2 = 0.30
  I3: Динамика дебит. задолженности (рост)   W3 = 0.20
  I4: Количество исторических нарушений      W4 = 0.15
  I5: Срок с последней проверки (давность)   W5 = 0.10

Пороговые значения:
  RS < 25      — НИЗКИЙ риск (зелёный)
  25 ≤ RS < 50 — СРЕДНИЙ риск (жёлтый)
  50 ≤ RS < 75 — ВЫСОКИЙ риск (оранжевый)
  RS ≥ 75      — КРИТИЧЕСКИЙ риск (красный) → автосигнал
```text

## ПРИЛОЖЕНИЕ Е ПОЛЬЗОВАТЕЛЬСКИЕ СЦЕНАРИИ (USER SCENARIOS)

### Е.1. Сценарий «Инспектор в полевых условиях»

1. **Подготовка**: Инспектор в офисе КРУ загружает на планшет кэш данных по в/ч 54321, включая историю нарушений и сметы.

1. **Работа в оффлайн**: В ходе проверки в части (в сейфовой комнате без связи) инспектор заполняет опросный лист, фотографирует акт инвентаризации. Данные сохраняются в IndexedDB браузера.

1. **Синхронизация**: При возвращении в штаб округа (появление VPN) инспектор нажимает «Синхронизировать». Система выявляет конфликт: Старший аудитор в офисе уже внес правку в заголовок акта.

1. **Разрешение конфликта**: Система отображает Diff-интерфейс. Инспектор выбирает «Объединить правки». Данные уходят в БД, формируется финальный черновик Акта ревизии.

### Е.2. Сценарий «Начальник КРУ: Анализ аномалии»

1. **Сигнал**: На дашборде Начальника КРУ загорается красный индикатор: «в/ч 11223 — Критический риск (RS: 88)».

1. **Детализация**: Начальник кликает по объекту. Система показывает: «Фактор риска: 4 тендера с единственным поставщиком за месяц».

1. **Проверка**: Начальник открывает вкладку «Доказательная база», видит ссылки на строки загруженных XLSX-файлов Казначейства.

1. **Решение**: Начальник прямо в системе назначает внеплановую дистанционную проверку Группе дистанционного аудита.
