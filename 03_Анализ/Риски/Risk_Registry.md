## Н.1. ОБЩИЕ ПОЛОЖЕНИЯ

### Н.1.1. Назначение

Настоящее приложение содержит реестр рисков проекта создания АИС КРУ МО РУЗ, включая идентификацию, оценку, стратегии реагирования и планы мониторинга.

### Н.1.2. Классификация рисков

| Категория | Описание | Примеры |
|-----------|---------|---------|
| **Технические** | Связаны с технологиями и архитектурой | Задержка поставки ПО, несовместимость |
| **Организационные** | Связаны с людьми и процессами | Задержка согласований, текучка |
| **Внешние** | Внешние факторы | Изменение законодательства, форс-мажор |
| **Ресурсные** | Связаны с бюджетом и персоналом | Превышение бюджета, нехватка специалистов |

### Н.1.3. Шкала оценки

**Вероятность:**

| Уровень | Балл | Описание |
|---------|------|---------|
| Очень высокая | 5 | > 80% вероятность |
| Высокая | 4 | 60-80% вероятность |
| Средняя | 3 | 40-60% вероятность |
| Низкая | 2 | 20-40% вероятность |
| Очень низкая | 1 | < 20% вероятность |

**Влияние:**

| Уровень | Балл | Описание |
|---------|------|---------|
| Критическое | 5 | Провал проекта |
| Высокое | 4 | Существенное влияние на сроки/бюджет |
| Среднее | 3 | Заметное влияние, преодолимое |
| Низкое | 2 | Минимальное влияние |
| Очень низкое | 1 | Пренебрежимое влияние |

**Рейтинг риска = Вероятность × Влияние:**

| Рейтинг | Диапазон | Цвет | Действие |
|---------|----------|------|---------|
| **Критический** | 20-25 | 🔴 | Немедленное устранение |
| **Высокий** | 12-19 | 🟠 | Приоритетное устранение |
| **Средний** | 6-11 | 🟡 | Плановое устранение |
| **Низкий** | 2-5 | 🟢 | Мониторинг |

## Н.3. ТОП-5 КРИТИЧЕСКИХ РИСКОВ

### Н.3.1. Риск T-01: Задержка поставки серверного оборудования

| Атрибут | Описание |
|----------|---------|
| **Описание** | Поставка серверного оборудования задерживается более чем на 4 недели |
| **Причина** | Логистические проблемы, дефицит на рынке |
| **Последствия** | Срыв сроков этапа «Разработка инфраструктуры» |
| **Вероятность** | 3 (Средняя) |
| **Влияние** | 4 (Высокое) |
| **Рейтинг** | 12 (Высокий) |
| **Меры митигации** | 1. Резервирование 2 недель в графике<br>2. Предварительное размещение заказа<br>3. Альтернативный поставщик |
| **План Б** | Использование облачной инфраструктуры как временного решения |
| **Ответственный** | DevOps |
| **Мониторинг** | Еженедельно |

### Н.3.2. Риск O-02: Изменение требований заказчиком

| Атрибут | Описание |
|----------|---------|
| **Описание** | Заказчик вносит существенные изменения в требования после утверждения ТЗ |
| **Причина** | Неполное понимание потребностей, изменение приоритетов |
| **Последствия** | Увеличение сроков и бюджета на 20-30% |
| **Вероятность** | 4 (Высокая) |
| **Влияние** | 4 (Высокое) |
| **Рейтинг** | 16 (Высокий) |
| **Меры митигации** | 1. Формализованный процесс изменений (RFC)<br>2. Привлечение заказчика к приёмке этапов<br>3. Прототипирование |
| **План Б** | Пересмотр контракта с учётом дополнительных работ |
| **Ответственный** | PM |
| **Мониторинг** | Еженедельно |

### Н.3.3. Риск R-01: Превышение бюджета

| Атрибут | Описание |
|----------|---------|
| **Описание** | Фактические затраты превышают утверждённый бюджет более чем на 10% |
| **Причина** | Недооценка сложности, дополнительные работы |
| **Последствия** | Необходимость дополнительного финансирования |
| **Вероятность** | 3 (Средняя) |
| **Влияние** | 4 (Высокое) |
| **Рейтинг** | 12 (Высокий) |
| **Меры митигации** | 1. Резерв бюджета 15%<br>2. Еженедельный контроль burn rate<br>3. Раннее выявление отклонений |
| **План Б** | Сокращение функционала второй очереди |
| **Ответственный** | PM |
| **Мониторинг** | Еженедельно |

## Н.4. СТРАТЕГИИ РЕАГИРОВАНИЯ НА РИСКИ

### Н.4.1. Стратегия снижения (Mitigation)

| Действие | Описание | Примеры |
|----------|---------|---------|
| **Предотвращение** | Исключение возможности наступления риска | Двойной поставщик |
| **Снижение вероятности** | Уменьшение шансов наступления | Обучение персонала |
| **Снижение влияния** | Уменьшение последствий | Резервное копирование |
| **Разделение риска** | Передача части риска третьей стороне | Страхование |

### Н.4.2. Стратегия принятия (Acceptance)

| Действие | Описание |
|----------|---------|
| **Активное принятие** | Создание резервов на покрытие риска |
| **Пассивное принятие** | Принятие без дополнительных мер (для низких рисков) |

### Н.4.3. Стратегия избегания (Avoidance)

| Действие | Описание |
|----------|---------|
| **Изменение плана** | Альтернативный подход, исключающий риск |
| **Отказ от работ** | Исключение рискованных работ из проекта |

## Н.5. ПЛАН МОНИТОРИНГА РИСКОВ

### Н.5.1. Регулярность мониторинга

| Тип риска | Частота | Участники |
|-----------|---------|----------|
| Критические (рейтинг >15) | Еженедельно | PM, Архитектор |
| Высокие (рейтинг 10-15) | Еженедельно | PM |
| Средние (рейтинг 5-10) | Ежемесячно | PM, Лиды |
| Низкие (рейтинг <5) | Ежеквартально | PM |

### Н.5.2. Реестр наблюдения (Watch List)

| Риск | Причина включения | Проверка | Ответственный |
|------|-----------------|----------|--------------|
| E-01 | Изменчивость законодательства | Новостной мониторинг | Юрист |
| E-05 | Рост кибератак | Ежедневный мониторинг SOC | Безопасник |
| T-04 | Сложность интеграций | Техническая экспертиза | Архитектор |

### Н.5.3. Триггеры эскалации

| Триггер | Значение | Действие |
|---------|---------|----------|
| Просрочка этапа | > 5 рабочих дней | Эскалация PM |
| Превышение бюджета | > 5% | Эскалация PM + Заказчик |
| Критический дефект | Severity: Critical | Немедленное совещание |
| Потеря ключевого специалиста | > 2 недели | HR action |
